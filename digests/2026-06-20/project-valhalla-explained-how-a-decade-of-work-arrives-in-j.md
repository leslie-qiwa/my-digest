# Project Valhalla, Explained: How a Decade of Work Arrives in JDK 28

Project Valhalla 详解：十年磨一剑，终现 JDK 28 - JVM 周刊第 180 期

新一期 JVM 周刊来了……诸神黄昏似乎即将降临，因为我们终于在 JDK 中迎来了 Valhalla。不过，情况有些……微妙。

6 月 15 日，Oracle 工程师 Lois Foltan 确认了业界很大一部分人早已不再相信的事情：JEP 401: Value Classes and Objects 将被集成到 OpenJDK 主仓库中，目标版本为 JDK 28。

这次变更规模之大，以至于其他提交者被要求在集成期间暂停较大的代码提交。仅这一个 Pull Request 就在 1,816 个文件中新增了超过 19.7 万行代码。

不过，在开香槟之前需要说明：这是一个预览特性，默认处于禁用状态，而且正如 Brian Goetz 急忙给大家降温时所说的那样，这"只是 Valhalla 的第一部分"。Goetz 还做了一个精妙的观察：那些"他们永远不会发布"的人，现在会无缝切换到"但他们没有发布最重要的部分"（社区里多年来一直流传着一个笑话：比起这个项目发布，我们自己先进入北欧神话中的英灵殿的可能性更大）。

拥有自己的黑粉，那是你挣来的。

所以现在是讲述整个故事的好时机。本期是一篇大型深度解读，写作时假设你之前从未关注过 Valhalla 的工作：从 2014 年的问题起源，到思路的演进（其中不少最终被丢进了废纸篓），一直到我们在 JDK 28 中具体能得到什么。给自己泡杯咖啡吧。我酝酿这一期已经很久了，就是为了等这个时刻。

1. 引言——这到底是怎么回事

Valhalla 从一开始就打出的口号是："像类一样编写，像 int 一样运行。"一句话就道出了整个项目的核心：我们想编写正常的、可读的类，带有方法、构造函数验证和合理的字段名称，但同时希望 JVM 能够像处理基本类型一样高效地处理它们。

要理解为什么这是个问题，必须回到 Java 的根基。在这门语言中，除了八种基本类型（int、long、double、boolean 等）之外，一切都是引用类型。当你写下 Point p = new Point(1, 2) 时，变量 p 并不是一个点。变量 p 是一个指针，一个衣帽间的号牌：堆上某个地方放着一个对象，而你手里拿着的是一张写有其地址的纸条。每次你想读取一个字段，JVM 都必须"去衣帽间取东西"，执行一次指针跳转（指针间接引用）。

对于单个对象来说，这不算什么。问题出在规模化时。堆上的每个对象都有自己的对象头（十几个字节的元数据：其中包括让 JVM 知道它是什么类型、是否有人正在对它进行同步等信息）。顺便说一句，这正是 Project Lilliput 最近一直在解决的问题，帮助缩小对象头的大小。但对象头大小并不是全部。每个对象都需要被分配内存，之后还要被垃圾回收。而且由于对象分散在堆上的各处，一个包含一百万个 Point 的数组实际上就是一百万张纸条，指向散布在整个仓库中的一百万个箱子。

Brian Goetz 在他的"Valhalla 现状"文档中，将这种内存布局称为"蓬松的"（fluffy）：膨胀的、臃肿的。我们梦想的是一种紧凑布局，数据紧挨着排列在一起。

为什么紧凑性很重要？因为硬件的进化速度远超 Java。在 1995 年，一次内存访问的开销与一次 CPU 运算大致相当。而今天，CPU 的速度比主存快了两个数量级，整个差距靠缓存来弥补。处理器以称为"缓存行"（cache line）的块为单位读取内存（通常为 64 字节）。如果数据紧密且有序地排列，一个这样的块就能一次性带入大量有用的值。如果我们在指针之间跳来跳去，每次访问都有缓存未命中的风险，而一次未命中可能比命中慢一百倍。这就是引用局部性（locality of reference），也是这整场博弈中真正的赌注。

"但 JVM 有逃逸分析啊，"有人会敏锐地指出。没错：虚拟机能够识别出某个对象从未"逃逸"出代码的某个局部片段，然后就根本不分配它。从程序员的角度来看，对象似乎存在，但实际上它的字段被展开成了普通变量或 CPU 寄存器。在最好的情况下，分配的开销和后续垃圾回收器的清理成本几乎降为零。

问题在于，这种优化是不可预测且脆弱的。它只有在 JIT 编译器能够高度确信地追踪对象的整个流向时才有效。但只要对象落入另一个类的字段中、被存入数组、被传入更复杂的方法，或者出现在 JIT 能分析的代码边界之外，整个技巧就失效了。源代码保持不变，但性能表现却可能发生巨大变化。

这正是经验丰富的 JVM 程序员将逃逸分析视为锦上添花而非项目基石的原因。如果应用的性能取决于某个特定 JIT 版本是否能应用这种优化，就很容易陷入难以预测的性能回退陷阱。一次小重构、一次 JDK 更新，或代码结构的改变，都可能让对象重新回到堆上，分配和垃圾回收器的开销又会全面恢复。

剩下的就是蛮力方案：放弃对象，手动编码数据。不用 Color 类，而是持有三个字节 r、g、b。这不仅仅是学术示例。这种方法多年来一直被用于游戏引擎、图形库、图像处理系统、数据库、分析引擎和 HPC 代码中，在这些场景下每个内存字节和每次分配都至关重要。问题在于，速度的提升是以牺牲安全性和可读性为代价的。我们失去了名称、私有状态、验证和方法。JEP 401 给出了一个简单的例子：处理"原始"颜色字节的开发人员可能会错误地将它们解读为 BGR 而不是 RGB，把红色和蓝色搞反，悄无声息地破坏整个图像。一个类不会允许这种情况发生。一个裸 int？当然会。

而正是这种二元对立——要么是方便的类，要么是高效的基本类型——Valhalla 试图消除的。

2. 起源——2014 年，"六个博士"和五个原型

Project Valhalla 于 2014 年正式启动。James Gosling 当时将其描述为"六个博士学位纠缠成一个结"，这并非夸张。有趣的是，这个想法比项目本身还要古老：Java 的创造者们早在语言的第一个版本时就想要值类型，但在 1995 年他们放弃了，因为这个问题太难了。

目标设定得雄心勃勃：恢复编程模型与现代硬件性能特征之间的一致性。换句话说，让程序员声明自己的类型，这些类型在内存中像基本类型一样扁平和紧凑，但看起来和行为上像普通的类。

说起来容易做起来难。在接下来的几年中，团队构建了五个不同的原型，每个原型都在探索问题的不同方面。这也是故事中最有趣的部分开始的地方，因为要欣赏 Valhalla 当前的形态，你必须看到有多少想法在过程中夭折了。

早期的原型走向了我们现在所说的"Q 世界"方向。它假设新的值类型与对象是根本不同的东西，有独立的类型描述符、独立的字节码和独立的顶层类型，就像基本类型一样。听起来合乎逻辑：如果它们应该像 int 一样工作，就让它们像 int 一样表示。问题在于，这种分离使整个 JVM 类型系统充斥着额外的复杂性：所有东西都必须做两个版本。

突破来自于一个被命名为"L 世界"的原型（大约在 2019 年左右）。这个名字源于值类型开始与普通引用共享相同的"L 载体"（L 描述符，JVM 用于普通引用的同一个）这一事实。
