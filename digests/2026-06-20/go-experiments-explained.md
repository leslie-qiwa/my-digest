# Go Experiments Explained

不确定如何组织你的 Go Web 应用程序结构？
我的新书将引导你从头到尾构建一个真实的 Go Web 应用程序——涵盖如何组织代码结构、管理依赖、创建动态数据库驱动页面，以及如何安全地进行用户认证和授权等主题。
来看看吧！Go 通常会在发布版本中附带实验性功能。
这些实验性功能可以采用不同的形式：有时是标准库中全新的包，有时是编译器或运行时的更改，或者——极少数情况下——可能是对 Go 行为的破坏性更改。
大多数情况下，实验性功能的目的是在某项特性正式发布并成为 Go 的永久组成部分之前，从用户那里获取真实的反馈。如果该功能导致了回归问题，或者收到了社区的负面反馈，那么在最终确定之前可以进行更改——甚至可以完全放弃。
## 一些示例
让我们看几个近期的示例，来说明 Go 实验可以涵盖的内容类型。
Go 1.24 附带了对新的 `testing/synctest` 包的实验性支持（该包提供了对并发代码测试的支持）。在收到反馈后，包的 API 进行了轻微调整，并在 Go 1.25 中正式发布。

Go 1.25 附带了对新垃圾回收器设计的实验性支持，具有更好的性能。在整合反馈后，新的垃圾回收器在 Go 1.26 中成为默认设置。

Go 1.21 附带了对循环变量语义的实验性行为更改。这一更改消除了 Go 代码中一个之前常见的 bug，但从技术上讲是对语言的破坏性更改。将该更改作为实验发布，给了人们在新行为成为 Go 1.22 默认设置之前测试其代码的机会。
## 实验生命周期
实验并没有一个固定的统一生命周期，但有一些常见的模式。
大多数实验最初以默认关闭的方式发布。你需要显式选择启用来试用该功能，通常通过设置 `GOEXPERIMENT` 环境变量（我们稍后会详细讨论）。
如果一切顺利，一到两个版本之后，实验性功能就会最终确定，正式发布，并成为默认开启。
如果实验影响了某些行为，那么在正式发布之后，有时——但并非总是——会有一个过渡宽限期，在此期间可以暂时禁用它并使用旧行为。例如，在 Go 1.26 中，新的垃圾回收器设计（我们前面简要提到过）正式发布并默认开启，但如果需要，仍然可以禁用它并使用旧的垃圾回收器。
这是最常见的模式，但有时事情需要更长时间或有不同的结果。例如：
Go 1.22 附带了编译器内联逻辑的实验性实现，两年多后仍然是默认关闭并在评估中。
同一版本还附带了内存竞技场（memory arenas）实验。在收到用户的负面反馈和担忧后，它仍然默认关闭，无限期搁置，最终可能会被完全移除。
或者最后，当 Go 团队对某个更改有信心时，他们可能会跳过反馈阶段直接正式发布……但仍然可能有一个过渡宽限期，在此期间可以禁用它。
一个很好的例子是 Go 1.24 将其 map 实现更改为使用 Swiss tables。Go 团队对该实现及其性能优势有足够的信心，使其直接正式发布并默认开启，但——至少目前——如果你想的话，仍然可以选择退出并使用旧的 map 实现。
所以在实践中，实验实际上有三种大的状态：
- 默认关闭且正在评估中
- 默认关闭且搁置/休眠
- 默认开启且提供临时退出选项
## 永久实验
Go 还有少量实验性功能，它们并不是通常意义上的"实验"。
这些功能默认关闭，但它们不在评估中，不寻求反馈，也没有预期它们会正式发布并默认开启。
虽然它们与其他实验一样通过 `GOEXPERIMENT` 环境设置来控制，但实际上它们更像是可选的 Go 功能，你可能会在特定场景下使用。
在本文的其余部分，我将把它们称为"永久实验"。
例如，有一个字段跟踪诊断功能，用于跟踪哪些结构体字段被访问。它已经存在了十年，没有任何意图让它正式发布。或者有一个静态锁排序功能，这是一个用于发现 Go 运行时中潜在死锁的诊断工具。
## 目前有哪些可用的实验？
要找出当前有哪些实验性功能可用以及它们的状态，出乎意料地困难。
不幸的是，官方 Go 文档或 Go Wiki 中没有一个页面来跟踪实验状态，对于本文，我不得不从各种地方拼凑信息。如果你想做同样的事：
你可以通过运行 `$ go doc goexperiment.Flags` 获取所有可用实验的列表。
你可以通过阅读 `src/internal/buildcfg/exp.go` 的源代码来弄清楚哪些实验是默认开启的——具体来说是查看 `ParseGOEXPERIMENT()` 函数中的 `baseline` 变量声明。
你可以将实验名称与 Go 发布说明交叉引用，并在 GitHub issues 中搜索以尝试弄清楚当前状态。
据我所知，截至 Go 1.26，以下是可用的永久实验：

| 实验名称 | 描述 | 状态 |
|---|---|---|
| `FieldTrack` | 跟踪哪些结构体字段被访问的诊断工具 | 默认关闭，永久保留 |
| `StaticLockRanking` | 验证锁获取顺序以捕获死锁的诊断工具 | 默认关闭，永久保留 |
| `CgoCheck2` | 检查 cgo 指针传递规则的诊断工具；运行开销太大，不适合默认开启 | 默认关闭，永久保留 |
| `BoringCrypto` | 用经过 FIPS 验证的 BoringSSL 替换 Go 的加密库；自 Go 1.24 起不再相关 | 默认关闭，永久保留，但即将移除 |
| `PreemptibleLoops` | 允许调度器在循环回边处抢占 goroutine；自 Go 1.14 起通常不再需要，但在其他方式不支持抢占的平台上可能仍然有用 | 默认关闭，永久保留 |

以下是当前默认关闭的实验及其状态：

| 实验名称 | 描述 | 状态 |
|---|---|---|
| `HeapMinimum512KiB` | 将最小堆大小从 4MB 减少到 512KiB；可能对受限环境有用 | 默认关闭，可能处于休眠状态 |
| `Arenas` | 内存竞技场实现 | 默认关闭，因负面反馈而搁置 |
| `NewInliner` | 重写的编译器内联器，具有更好的调用点启发式算法 | 默认关闭，评估中（自 Go 1.22 起可用） |
| `JSONv2` | 新的 `encoding/json/v2` 包，提供改进的 JSON 编解码函数 | 默认关闭，评估中（自 Go 1.25 起可用） |
| `RuntimeSecret` | 新的 `runtime/secret` 包，提供清零内存的函数；仅在 Linux amd64/arm64 上可用 | 默认关闭，评估中（自 Go 1.26 起可用） |
| `GoroutineLeakProfile` | 添加 goroutineleak pprof 分析类型 | 默认关闭，评估中（自 Go 1.26 起可用） |
| `SIMD` | 新的 `simd/archsimd` 包，提供对架构特定 SIMD 操作的访问；仅在 amd64 上可用 | 默认关闭，评估中（自 Go 1.26 起可用） |
| `RuntimeFreegc` | 允许在安全的情况下立即重用内存，而无需等待 GC 周期 | 默认关闭，评估中（自 Go 1.26 起可用，但请参阅 #74299 了解状态信息） |
