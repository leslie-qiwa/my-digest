# How Go Reflection Really Works

在上一篇文章中，我们观察了运行时如何利用编译器和链接器在构建时冻结到二进制文件中的元数据，重建出完整的栈追踪。在文章结尾我告诉过你，reflect 的工作原理完全相同——元数据被烘焙进二进制文件，只不过指向的是你的数据而非调用栈。今天我们来兑现这个承诺。

让我们从一个程序开始，第一次看到它时，你会觉得这根本不可能：

```go
package main

import (
    "fmt"
    "reflect"
)

type User struct {
    Name  string `json:"name"`
    Email string `json:"email,omitempty"`
    age   int
}

func main() {
    t := reflect.TypeOf(User{})
    for i := 0; i < t.NumField(); i++ {
        f := t.Field(i)
        fmt.Printf("%-6s %-8s %q\n", f.Name, f.Type, f.Tag)
    }
}
```

运行它，输出如下：

```
Name   string   "json:\"name\""
Email  string   "json:\"email,omitempty\""
age    int      ""
```

字段名。字段类型。甚至结构体标签，一字不差。在运行时。

现在仔细体会这有多奇怪。Go 是静态类型语言，编译为原生机器码。没有虚拟机在维护类对象，没有解释器还持有你的源代码。当这个程序运行时，`User` 本应已经消失——被压缩为内存中 40 个字节，仅此而已。`Name`、`Email` 和 `age` 这些名字是你写给人看的；CPU 从不需要它们。那么这些信息从何而来？

这就是今天的核心问题，答案有着令人满意的结构：`reflect` 包并不计算这些信息。它只是读取。真正的工作发生在数周之前，在你的机器上，当你输入 `go build` 的时候。

> 关于范围的说明：本文聚焦于 `reflect` 的读取侧——检查类型和值。这个包还能在运行时构建东西——新的结构体、函数值、切片等等——那是另一个完整的故事，我可能会在未来的文章中再来讨论。

但这样的断言需要对源码进行一次正式的追踪——所以让我们从表面开始，逐层深入。

## 入口：TypeOf 和 ValueOf

我们大部分的探索从两扇门之一开始：`reflect.TypeOf` 给你一个 `reflect.Type` 来查询类型信息，`reflect.ValueOf` 给你一个 `reflect.Value` 来检查（有时修改）一个实际的值。两者都接受 `any` 作为参数。你会以为像反射这样强大的功能，其入口点应该是魔法发生的地方——所以让我们看看 `TypeOf`：

```go
func TypeOf(i any) Type {
    return toType(abi.TypeOf(i))
}
```

嗯，再往下一层。这是 `abi.TypeOf`：

```go
func TypeOf(a any) *Type {
    eface := *(*EmptyInterface)(unsafe.Pointer(&a))
    return (*Type)(NoEscape(unsafe.Pointer(eface.Type)))
}
```

就这样了。没有表查找，没有运行时调用，没有内存分配。取 `any` 参数的地址，将那里的内存重新解释为一个名为 `EmptyInterface` 的结构体，然后返回它的第一个字段。`reflect.TypeOf` 归根结底就是一次指针加载。

这只能说明一件事：在 `reflect` 介入之前，类型的完整描述就已经在 `any` 里面了。有趣的不是这个函数——而是 `any` 到底是什么。

## 接口的内部结构

以下是 `EmptyInterface` 结构体的样子（`src/internal/abi/iface.go`）：

```go
type EmptyInterface struct {
    Type *Type
    Data unsafe.Pointer
}
```

你程序中的每一个 `any`——每一个空接口值——都恰好是这样：两个指针。第二个指向数据。第一个指向一个类型描述符：一个完整描述了存储在接口中的动态类型的结构体（我们稍后会深入了解）。运行时有自己的镜像布局，叫做 `eface`。

所以当你把 `user` 传给一个接受 `any` 的函数时——这恰好就是调用 `TypeOf` 时发生的事情：

```go
func TypeOf(i any) Type // 我们刚看到的签名
TypeOf(user)            // user 在这里变成了 any
```

编译器在调用点生成的代码存储了两个指针：`User` 类型描述符的地址和一个指向值的指针。这基本上就是整个转换过程——数据指针只是指向内存中某处的值（在栈上或堆上），但类型指针不涉及任何查找和注册：它是编译器在编译时就已知的常量地址。

这完全重新定义了我们对这两个入口点的理解。`TypeOf` 和 `ValueOf` 根本不检查你的值。它们读取的是接口头——编译器早已放在那里的两个指针。

但这只是把问题往后推了一步。类型指针指向一个知道字段名和结构体标签的描述符。谁构建了那个描述符，它存放在哪里？

## 编译器干的

这是我觉得真正优美的部分：那个描述符是静态数据。编译器构建了它，链接器放置了它，它就在你的可执行文件的只读段中，在你的程序执行第一条指令之前就已经在那里了。

当编译器处理你的包时，一个叫做 `reflectdata` 的组件充当类型的序列化器。对于每一个可能出现在接口中的类型，它都会生成一个名为 `type:<你的类型>` 的符号——`type:main.User`、`type:[]string`、`type:map[string]int`——其原始字节就是描述符结构体，逐字段排列（`writeType`）。

你不必只是相信我的话。编译文章开头的程序并对其运行 `strings`：

```bash
$ go build -o hello .
$ strings hello | grep omitempty
json:"email,omitempty"
```

你的结构体标签就在那里，一字不差，作为可执行文件中的字节。它从未"在运行时被保留"——它是被编译进去的，就像字符串字面量一样。反射最深层的秘密是根本没有什么运行时机制可言：它就是编译器给未来的自己写了一些笔记，而 `reflect` 用指针算术把它们读了回来。

不过到目前为止，我们把每个描述符都当成了一个不透明的二进制块——要理解 `reflect` 如何在其中找到字段名，我们需要深入到字节层面。

## 类型数据的形状

每个描述符是一段连续的只读字节，编译器塑造这段字节的方式是整个设计的精妙之处的一半。它由多层构成：

第一层对于二进制文件中的每一个类型都是相同的，无论它是 `int`、channel 还是你的 `User`：一个紧凑的头部，记录了类型的大小和对齐方式、它的种类、一个预计算的哈希值使得类型 switch 永远不需要重新计算、垃圾回收器遍历的指针位掩码、`==` 的实际实现以及类型的名称。我不会在这里逐字段展开——如果你想更深入了解本文所有内容背后的数据结构，请查看 `internal/abi` 中的 `Type` 结构体，这个包充当编译器（写入字节）与运行时和 `reflect`（读取字节）之间的契约。

现在，这个头部是固定大小的，但每种类型需要不同的额外信息：结构体有字段列表，函数有参数和返回值，切片只需要其元素类型。所有这些信息放在哪里？它不能放在头部内部——固定大小的结构体无法容纳变长数据——而在每个描述符中为所有可能性预留空间，会在大多数类型永远用不到的字段上浪费空间。

编译器的解决方案是 C 程序员一眼就能认出的技巧：将特定种类的数据直接写在头部之后，紧接着的字节中。结构体描述符是通用头部后跟它的字段数组；切片描述符是相同的头部后跟它的元素类型；函数描述符是头部后跟它的参数和返回值。每个描述符都以相同的方式开始；后面跟着什么取决于种类。

读取端与写入端镜像对应。当 `reflect` 看到头部的种类字段显示为 `Struct` 时，它就知道后续字节的形状，并将整个区域转换为 `abi.StructType`——一个以通用头部开始的结构体：

```go
type StructType struct {
    Type    // 通用头部，嵌入在偏移量 0 处
    PkgPath Name
    Fields  []StructField
}

type StructField struct {
    Name Name // 名称和标签，
```
