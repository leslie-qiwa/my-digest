# I Stored a Website in a Favicon

前阵子我写过一篇文章，讲的是在鼠标的 DPI 寄存器里存储两个字节。那没什么用，也不实际。但它对我的大脑产生了一种不幸的影响。一旦你成功地把数据藏在了一个不该存数据的地方，你就会开始把所有东西都当成潜在的存储介质。

显示器是存储。

键盘是存储。

BIOS 启动画面（也许）是存储。

网站图标是存储。

是的，我们来了。

每个网站都有一个 favicon（网站图标）。就是浏览器标签页上的那个小图标。通常你上传一次之后就再也不会想起它了。但是。favicon 只是一张图片。图片只是像素。而像素只是字节。

所以我自然而然地想知道，能不能在里面存点什么。

## 想法

我最初想到的是隐写术。

隐写术基本上就是在不让人察觉的情况下把数据藏进图片里。你拿一张完全正常的照片，修改几个比特位，让它秘密地包含一段信息。

favicon 本身（至少在我的演示中）不需要看起来像一个图标。它可以变成纯粹的存储。

每个像素都有红、绿、蓝三个值。也就是三个字节。如果我想存储文本，我可以直接把文本的 UTF-8 字节写入 RGB 通道。

浏览器不在乎这些字节代表什么。对浏览器来说它们是颜色。对我来说，在这个场景里，它们是 HTML。

## 构建一个 favicon 网站

我从一个极小的 HTML 载荷开始：

```
<h1>Website in a Favicon</h1>
<p>
Everything you're reading right now was decoded from favicon pixels.
</p>
```

整个过程非常简单直接。

首先，我用 `TextEncoder` 把 HTML 转换为字节。

然后在前面加上四个字节，包含载荷的长度。

长度头很重要，因为图片本身末尾可能包含未使用的像素。如果没有长度值，就没办法知道真正的载荷在哪里结束。

接着我开始填充像素：第一个字节变成第一个像素的红色通道，第二个字节变成绿色通道，第三个字节变成蓝色通道，然后是下一个像素，再下一个，再下一个，直到整个 HTML 文档以彩色像素的形式存在。结果看起来像视觉噪点。

## 非常小

说实话，最让我惊讶的不是它能跑通，而是生成的图片有多小。

载荷最终是 208 字节。

加上 4 字节的头，总共 212 字节。

由于每个像素存储三个字节，我需要：

- 总共 212 字节
- 71 个像素
- 一个足够大的正方形图片来容纳它们

最小可用的正方形是 9x9 像素。

只有 81 个像素。

最终的数据如下：

- 载荷：208 字节
- 图片尺寸：9x9 像素
- 容量：239 字节
- 使用率：87%

一个完整的小网站（好吧，是带一些样式的 HTML）竟然塞进了一张比通常的 favicon 还小的图片里。

## 把网站读回来

存储数据只是问题的一半。另一半是把它取出来。

浏览器已经具备了所需的一切。

- favicon 作为图片被加载。
- 图片被绘制到 canvas 上。
- Canvas API 允许 JavaScript 读取每个像素。

一旦我拿到了像素数据，只需反转整个过程。

- 读取 RGB 值。
- 重建字节数组。
- 读取前四个字节以确定载荷长度。
- 提取载荷。
- 解码 UTF-8 文本。

这时我就拿回了原始的 HTML。

浏览器从自己的 favicon 中读出了一个网站。

## 重要的注意事项

favicon 实际上并不包含整个网站本身。

它包含的是网站的内容。

你仍然需要一个微型的引导加载器来解码图片。

没有 JavaScript 的话，favicon 只是一个 PNG（里面包含着你的网站内容）。

为了展示这个场景，网站包含了一个"渲染网站"按钮。它读取 favicon，解码 HTML，然后用重建的内容替换页面。

## 这有用吗？

不，当然没有。

能存储的数据量微乎其微。页面需要 JavaScript 来引导自身。有几十种更好的方式来分发一个小型 HTML 文档。

但归根结底，这不就是为了测试边界吗？

favicon 感觉是一个非常特定的东西。它应该是一个图标。

但归根结底，它只是一个 PNG。

而 PNG 文件基本上就是字节。

这大概是我建过的最小的网站了……

## 替代方案

- 将标记直接存储在 SVG favicon 中，在页面加载时读取。
- 使用 PNG 注释块，如 tEXt、zTXt 和 iTXt。
- 使用 ico 文件格式，因为它允许包含多个不同分辨率的图标。
