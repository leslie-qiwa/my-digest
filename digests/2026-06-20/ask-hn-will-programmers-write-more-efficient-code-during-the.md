# Ask HN: Will programmers write more efficient code during the memory shortage?

- 每个人都假设自己的程序/网站是某一时刻机器上唯一运行的东西，而且开发机器总是比用户机器更强大
- 导致臃肿的大多数时候并不是缺乏高级数据结构和算法，而是程序和网站由大型团队交付，有几十个子模块经常在不需要时也被加载，而要正确地架构以避免这种情况很难，不这样做就会陷入巨大的复杂性和潜伏的棘手bug——当其他团队的人修改了某些东西却不了解全貌时就会触发。所以保持现状以降低架构复杂性和脆弱性反而更便宜。

内存使用优化的主要场所将是服务器端，因为客户端的内存是一个公地悲剧。如果你在减少内存使用而你的竞争对手在增加功能，那么你的应用节省的额外内存只会被悄悄地分配给他们的应用，设备不会有任何不同的感觉，用户会更喜欢你的竞争对手。优化客户端内存使用几乎没有动力，因为它只会帮助其他公司，所以没人这么做——除了（讽刺的是）浏览器开发者，他们往往认为自己对设备的所有内存有优先使用权。

如果你真的想让人们关心这个问题，操作系统开发者需要以普通用户能理解的方式可视化地展示应用的内存使用情况，并将其转化为客户反馈表单，这很难做到。

> 内存使用优化的主要场所将是服务器端

同意，但这将是一场漫长、艰难、最终可能不成功的上坡之战。我认为主要障碍基本上是二十年的惯性。几乎所有人都相信服务器端软件需要水平扩展。所以我们针对这个特性来优化系统——我可以通过投入更多计算机来让它运行得更好。这在大约2004年（MapReduce时代）比2026年重要得多。为了弹性和冗余，你的服务实际上需要多少个节点？如果你运行的节点数超过了那个数量，你有什么理由来解释为什么吗？这是我们需要努力转向的思维方式，但它与我们目前的做法非常不同。

编辑：另一个我们做得不够的好问题是："它真的需要是一个独立的服务吗？"使用已经分配的内存中的数据可能比分配更多内存并通过网络传输数据要好……

> 程序和网站由大型团队交付，有几十个子模块经常在不需要时也被加载

正是如此。我们曾经遇到过一个事故，一个CSV中的邮政编码字段被pandas解释为整数，这当然会去掉任何前导零。在查看代码实际需要做什么之后，我问他们为什么要用pandas，因为它实际上只是"按原样将CSV读入一个列表"。猜猜Python标准库的csv模块不会做什么？类型推断。

他们没有用标准库替换不必要的pandas导入（它带来了相当重的传递依赖链），而是添加了额外的代码和测试来确保这种情况以后不会发生。

有些开发者通过尝试来学习。有些通过阅读文档来学习。大多数似乎是通过阅读使用不必要第三方包的博客文章来学习。

我天真地以为避免被呼叫（值班告警）就是足够的动力，但经验让我放弃了这个想法。一次又一次，"我们再修一个我们那个粗糙的、定制的破烂上的问题"战胜了"我们将使用无聊的、现成的解决方案"。我最喜欢的例子是一个团队用NodeJS为Postgres构建了一个负载均衡器/健康检查器。尽管大约每月导致一次事故，我始终无法说服任何人放弃他们的特殊宝贝。

老实说我不理解把这种东西写在简历上有什么用。如果我在面试中看到这个，我的第一个问题会是"你为什么要自己构建这个而不是使用HAProxy / AWS NLB等？"

我对pandas没有意见（虽然polars在需要的场景下快得多），就像我对numpy没有意见一样；我不喜欢的是人们为了最琐碎的、标准库已经解决的事情去使用它们。更令人恼火的是，当他们引用性能作为必要的理由，但实际上只是在循环中传递数据，几乎抵消了所有潜在的性能提升。

我要说这里还有一个因素，而且它可能是网站和Web应用最大的因素。

广告和追踪。

大多数大公司网站上大约90%的臃肿来自这些脚本被到处添加。理想情况下移除这些会使网站高效得多，但市场营销和销售人员可能不会允许。

这确实是可能的，但我同意许多其他评论者的看法，它可能不会发生。

我写了一个运行在普通操作系统上的加密网状网络库。一个客户问我能否让它在只有520 KiB内存的ESP32上运行。起初这似乎不可能，但事实证明是可以的，而且甚至不是很难。虽然原始库对于桌面CPU来说根本不算内存消耗大，但它仍然在不必要的大缓冲区上浪费了空间。去掉这些后，库就能在ESP32上运行，同时还留有充足的空间给应用程序。

另外，我的第一台电脑是一台200 MHz单核32位AMD K6，配有32 MiB内存。它运行着带有浏览器、文字处理器、3D游戏等的图形操作系统。如今你可以买到仅内置缓存就超过这个内存量的CPU。

所以优化代码的一个好起点是实际获取一台"严重资源受限"的计算机，然后让你的代码在上面运行。

一些程序员会编写更高效的代码。在我的工作单位（一家大型科技公司），我们已经在计划明年的一个主要目标——优化服务器代码以降低内存需求，这直接是对资源紧缩的回应。

实际上我预计大多数优化将来自"停止做蠢事"而不是"使用高级算法"。但这是一个愤世嫉俗的观点，所以不要像我一样愤世嫉俗。

确实如此，有太多唾手可得的优化机会但没人有时间去做。而且他们也不激励你花时间在这上面。

如果CEO们说的是真的，那么谷歌75%的新代码是AI生成的。GitHub内部代码的46%，微软全部代码的大约30%是AI生成的。Meta预计至少65%，Snap报告65%是AI生成的。

在AWS，一些核心的基础服务已经被新的服务架构替换了。那个东西基本上是用大语言模型写的。

请注意，那个人是杰出工程师，他的团队基本上都是首席工程师级别的，但你确实可以这么做，一些新团队正在模仿这种风格（尽管成功率较低，因为技术能力不足）。

这里的关键词是"服务器"。在服务器端优化内存直接转化为公司的成本节省。优化运行在用户计算机上的代码的内存使用，动力要小得多。

> 在服务器端优化内存直接转化为公司的成本节省。

前提是这些成本节省超过开发成本。优化工作通常由最有经验、因此也最昂贵的工程师完成。优化努力也可能无法产生有意义的结果。在我的职业生涯中，我见过的优化努力失败的次数多于成功的。

这是一个聪明的观察，而且非常正确。不过历史上开发时间也非常昂贵，因此才推动了容易使用的开发技术栈，比如js。

不过通常很难直接归因于用户流失，而某个巨大的EC2实例每天花费你30美元是一个清晰的费用项目，成本节省可以轻松衡量。
