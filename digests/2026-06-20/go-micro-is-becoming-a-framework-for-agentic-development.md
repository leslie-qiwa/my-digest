# Go Micro is Becoming a Framework for Agentic Development

2026年6月17日 • Asim Aslam

当 Anthropic 开始赞助 Go Micro 时，我们承诺以开放的方式构建并实时报告进展。至今已过去三个月。之前的每篇文章都只涉及一个变更；这一篇则是关于它们汇聚在一起意味着什么。Go Micro 正在成为一个面向智能体开发的框架，就像过去十年它一直是服务开发框架一样。这是现有框架的延伸，而非重写，因为智能体本身就是一个分布式系统，而 Go Micro 正是用来构建分布式系统的。

十年前，Go Micro 将分布式系统中的难题（服务发现、传输、编码、发布/订阅、存储）封装在一组可插拔的接口背后，并提供可用的默认实现，让你能够快速构建服务系统。本季度的前提是：智能体并非特例。智能体由模型、提示词和一组工具组成，一旦它拥有不止一个组件，它就再次成为一个分布式系统：它需要发现服务、调用服务、持久化状态并从故障中恢复。构建这些正是 Go Micro 的用武之地。

因此，我们的工作是让智能体成为同一框架中的一等公民，并明确边界。Go Micro 是基础设施，而非大脑。它不会在提示词工程上超越专门的提示词库。它提供的是让你将智能体和服务一起构建成系统的方式。（《微服务的演进》一文完整阐述了这一观点。）

值得注意的模式是：每个组件几乎不需要从零发明什么。

**智能体即服务。** `micro.NewAgent` 创建的智能体就是一个服务：它将所分配的服务发现为工具，运行模型的工具调用循环，并像其他服务一样注册一个 `Chat` RPC 端点。没有独立的运行时，也没有图 DSL；工具调用就是 RPC，由大语言模型驱动。（《服务的智能体》一文阐述了这个模型。）

**规划与委派。** 每个智能体都有两个内置工具：`plan`——一个有序的计划，持久化到其记忆中，并在后续轮次中回显给它；以及 `delegate`——通过 RPC 将子任务交给另一个已注册的智能体，或在没有专门智能体时交给一个专注的临时子智能体。多智能体系统由工具构建，而非由协调引擎驱动。

**工作流。** 并非所有任务都应由模型驱动。遵循 Anthropic 的分类体系，Go Micro 增加了确定性的对应物：`Flow` 是由事件触发的预定义路径。它可以自行执行某个步骤，也可以将事件交给智能体处理：工作流负责触发，智能体负责推理。

**护栏。** 自主智能体会以常见的方式失败：它们会循环、失控、执行不该执行的操作。三个护栏设置在每个工具调用必经的唯一节点上：`MaxSteps`、`LoopLimit`（默认开启）以及用于人机协作或策略控制的 `ApproveTool`。编排留在智能体内部；执行安全是一个独立的、可替换的关注点。

**工具执行中间件。** 因为工具调用就是 RPC，Go Micro 一直以来为客户端和服务端提供的中间件现在同样适用于工具。`AgentWrapTool` 包装执行过程以实现日志记录、指标采集或重试，采用与 `client.CallWrapper` 相同的 `func(next) next` 签名。护栏本身在底层也是包装器。

**持久化工作流。** 调用真实服务的工作流会在执行过程中产生副作用，中途崩溃时不应重复执行已完成的步骤。Flow 演变为有序的、带检查点的步骤序列，能从中断处恢复，底层由可插拔的 `Checkpoint` 支持（默认基于存储，也可委托给 Temporal 或 Restate）。这是"存储加接口"形式的持久化执行，无需运行额外的引擎。

**支付。** 一个自主行动的智能体最终需要为某些事情付费。Go Micro 支持 x402 协议：工具可以要求稳定币支付，智能体可以完成结算，链的选择通过可插拔的中介者实现。

**一致的状态模型。** 最近，底层管道实现了一致性。服务、智能体和工作流各自通过新的 `store.Scope` 句柄将状态保存在自己的存储表中（`service/{name}`、`agent/{name}`、`flow/{name}`），而不是修改一个全局存储。工作流像智能体一样在注册中心注册，因此 `micro flow list` 和 `micro agent list` 是按类型过滤的同一个命令，而 `micro flow runs` 和 `micro agent history` 读取持久化状态。实时状态来自注册中心，历史记录来自存储。

这一切都不是在原有框架上拼接的新框架，而是同一个框架指向了智能体：

没有需要学习的图，也没有需要在服务旁边运行的引擎。抽象仍然是服务；智能体是你现在用它来构建的东西。

在笔记本电脑上执行 `micro call` 是可以的，但当自主智能体可以访问端点时就不行了。v6 的主题是在不丢失零配置开发循环的前提下翻转这些默认值（TLS、身份认证、有界执行）。`Checkpoint` 应该让长时间运行的智能体在重启后存活并恢复。剩下的唯一结构性变更是让智能体拥有自己的循环。

一个小型的端到端测试工具现在可以启动一个真实环境（服务、一个会崩溃并恢复的持久化工作流，以及一个带护栏的智能体），验证结果，然后关闭。它在每次变更时使用模拟模型运行，并按计划使用真实模型运行。

最终结论是：在 Go Micro 中构建智能体看起来就像构建服务，因为在底层它就是服务。智能体系统的基础设施在很大程度上就是服务系统的基础设施，而这正是框架中已有的十年积累。服务依然是根基；智能体是我们在其之上构建的东西。
