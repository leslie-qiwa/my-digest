首先，有一件事稍微令人担忧：我们上一篇博客文章中的内容在 2026 年几乎仍然全部适用——但有一个重要的例外。

虽然路径遍历仍然是一个非常现实的问题，但一年多前发布的 Go 1.24 引入了一组名为 `os.Root` 的新 API。

这里不再过多重复 Go 团队博客文章中已经解释得很清楚的内容，这些 API 旨在通过创建一个根目录并在框架层面禁止超出此边界的读写操作，从而彻底阻止路径遍历。根目录以下的所有内容仍然可用，包括子目录和符号链接（只要它们保持在根目录范围内）。

虽然这是一个非常有用的功能，但它并非没有初期问题；不过这些问题在 Go 1.25 中很快得到了修复。

类似地，Go 团队刚刚完成了他们的 `math/rand` 改进工程，这项工作从 Go 1.20 持续到 1.26（即发布时的当前版本）。

Go 从"向 `rsa.GenerateKey` 传递 nil 会导致灾难性的静默错误"演变为"无论你传入什么，函数都会做正确的事情。"

这是通过对 `math/rand` 进行整理来实现的，使开发者可以在 PRNG 和 `crypto/rand` 之间做出明确的选择——而不是使用手动设置种子的生成器，这种方式在该库中已经完全消失了。

此外，在生成私钥时意外使用不安全随机数的暗坑也已被消除——方法是确保这些函数忽略传递给它们的任何 `io.Reader`，无条件地使用安全的内部随机数。

与大多数其他语言不同，Go 的 `int` 和 `uint` 类型没有固定宽度——它们在编译时根据目标平台设置：在 amd64 和 arm64 上为 64 位，在 386、arm 和 WebAssembly 上为 32 位。

这是为了做出高效的大小选择而做的设计决策；然而，由于 Go 支持交叉编译，这意味着除非开发者足够谨慎，否则可能会对整数的大小做出错误的假设。

Go 在整数溢出和下溢时不会产生任何运行时错误或 panic，而是静默允许，这在开发者未加注意时可能引入微妙且严重的安全漏洞。

举例来说，`strconv.Atoi` 这个经典的字符串转整数函数返回一个普通的 `int`，这意味着它的范围在不同架构上是不同的。

虽然 `strconv.Atoi` 在返回的 `int` 超出其范围时会报错，但这仍然可能在下游导致静默的回绕问题。

Paul Gerste 在 DEF CON 32 上的演讲"SQL 注入并未消亡：在协议层走私查询"（SQL Injection Isn't Dead: Smuggling Queries at the Protocol Level）展示了这一问题的实际影响。

Gerste 在 Go 数据库驱动库中发现了多个整数溢出漏洞，这些漏洞使得攻击者能够溢出相应线路协议的长度字段。

通过溢出长度字段，他随后演示了在数据包发送到后端数据库时注入新查询的攻击方式。

这类漏洞并非仅限于数据库驱动。

与互联网世界中几乎所有事物一样，HTTP 头部由几个 RFC 定义——不同实现对这些标准的遵守程度则是另一回事了。

与我们当前关注点相关的 RFC 之一是 RFC 2616，它定义了一组逐跳（hop-by-hop）头部。

顾名思义，这些头部仅对单个连接有意义，因此不应被代理转发。

附带说明一下，虽然该 RFC 现已过时并被下文描述的 RFC 所取代，但仍有遗留代码在遵守这一标准。

为了使事情"更简单"，有一个更新的 RFC——RFC 7230，旨在取代 RFC 2616——它定义了 `Connection:` 头部，该头部应向下一跳接收方发出信号，指明哪些头部是逐跳的，因此在转发消息之前应被移除。
