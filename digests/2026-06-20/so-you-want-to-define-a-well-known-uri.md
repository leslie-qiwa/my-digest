# 所以你想定义一个 Well-Known URI

2026年6月19日，星期五

作为 Well-Known URI 规范的作者之一，同时也是该注册表的现任指定专家，我经常收到关于如何使用它们的问题，也经常指导大家如何最好地使用它们。下面，我总结了我对这个问题的看法。请注意，这些并不全是注册的硬性要求——只是我认为的良好实践。

## Well-Known 位置适合做什么

Well-Known 位置最适合的场景是：客户端——无论是浏览器、爬虫还是其他软件——已经知道站点<sup>1</sup>，并且需要以高效的方式发现关于整个站点的某些信息。

robots.txt 就是一个完美的例子——它早于 RFC 出现，所以没有使用 well-known 位置，但它是我们为其预留空间的一个主要原因。爬虫需要了解站点的访问策略，将其放在站点的一个中心位置，避免了在每个响应上检查头部和内容的需要（这会违背制定此类策略的许多目的）。

不过，well-known 位置不一定要包含策略。任何客户端已经知道站点但需要了解该站点整体信息或与之交互的机制，都是成为 well-known 位置的候选。例如，`change-password` well-known 位置允许客户端更改其在某个站点上的密码。

## 什么时候它们不是合适的工具

虽然 well-known 位置确实为某些协议解决了实际问题，但在其他情况下，设计者似乎只是因为觉得这是应该做的事情而指定一个 well-known 位置。一些提案注册一个 well-known 位置，期望它能赋予合法性或促进采用——仿佛注册表中的一个槽位就是一种凭证。事实并非如此。Well-known 位置解决的是一个特定问题（客户端知道站点，并且需要站点范围的某些东西）；如果你的协议没有这个问题，注册可能只会制造新的问题——也不会带来你期望的采用率。

类似地，一些 well-known 位置的提案实际上是把它们当作 URL 缩短器来使用。它们不在协议中传递完整的 URL，而是只需传递相关的站点——well-known 位置填充其余部分。

问题在于，这种模式将你锁定在服务与站点之间的一对一关系中。如果某个部署需要多个服务，他们就需要创建不同的站点，并找到一种方式将用户引导到合适的站点。

如果你的协议确实只能携带主机名，使用 well-known 位置是合理的。但通常情况下，这样做只是为了方便——也许还为了让协议看起来更"正式"——在部署中造成了不必要的僵化。如果你的协议可以使用真正的 URL，就不要使用 well-known 位置。

## 常见陷阱与权衡

即使 well-known 位置是正确的工具，我们对站点所做的假设也不总是成立的，可能会造成显著的复杂性。如果你正在为协议定义一个 well-known 位置，你应该了解以下问题。

### 发现机制

许多协议试图将 well-known 位置用作发现机制，其想法是"用户已经知道站点"。

问题在于，现实比乍听起来要模糊得多——用户当前交互的范围与发现发生的位置之间可能存在不匹配。例如，如果客户端从 "login.example.com" 开始，它们应该在该站点上查找 well-known 位置，还是在 "example.com" 上查找？它们是否应该跟随从一个到另一个的重定向？发布者应该在哪些站点上提供 well-known 位置以确保互操作性？

这在协议并非真正关于 Web 站点，而只是利用 HTTP 来完成其他事情的情况下尤为重要。例如，可能很容易想到为可注册域名指定一个位于顶级域的 well-known 位置，但这在操作上对某些人来说可能很困难。

如果你的协议属于这一类别，请同时考虑被发现的是什么以及你的用户从什么开始，然后在不对其架构做过多假设的情况下，找出他们如何可靠地找到正确的主机名。

### 内容元数据

一些协议试图使用 well-known 位置来了解站点上的内容。毕竟，`/robots.txt` 就是这样工作的。

虽然这种模式对某些类型的元数据有效，但许多站点代表多个发布者（例如，旧的 `/~username/` 约定）。将内容元数据放在一个中心位置，要么使那些用户无法使用该机制，要么要求管理员开发复杂的基础设施来支持和监督他们对其的控制。

这类部署揭示了便利性与粒度之间的权衡，通常需要创建一个并行的元数据机制——例如，在 HTTP 响应头或内容本身中——以及对不同元数据附加机制的合理化。

这并不是说 well-known 位置不能用于内容元数据；只是它们比你想象的要复杂得多。因此，如果你的协议使用 well-known 位置来处理资源元数据，请确保你仔细考虑这些权衡。不是所有站点都像你习惯的那些——Web 是一个很大的地方。

### 其他注意事项

还有一些经常出现的其他问题。

首先，一些提案已经在根目录定义了固定位置（如 `/robots.txt`——通常是因为他们后来才了解到 Well-Known 位置）。如果是这样，他们应该有一个针对现有部署的过渡计划。通常，提案者过于关注他们当前的部署规模；通过一个合理时间段内的良好过渡计划，迁移到 Well-known 位置是可以管理的。

许多提案隐式地假设使用 `http` 和 `https` URL。Well-known 位置也适用于其他 URL 方案，因此应该明确列举相关的方案。

最后，确保你注册了你的 well-known 位置。该链接包含关于何时注册以及如何选择名称的指导；与这里的建议不同，它确实会影响你成功注册的机会。
