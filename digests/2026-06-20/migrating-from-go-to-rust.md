# Migrating from Go to Rust

在我帮助团队进行的所有迁移中，从 Go 到 Rust 的迁移算是比较特殊的。这不是"Rust 是不是更快？"或者"Rust 有没有类型？"的问题——Go 在这些方面已经做得很好了。讨论的焦点主要集中在正确性保证、运行时权衡和开发者体验上。

在开始之前先做一个简短的声明：本指南侧重于后端领域。后端服务是 Go 最擅长的地方——小巧的静态二进制文件、专注于网络的标准库，以及丰富的 HTTP 服务器、gRPC、数据库等库生态系统。

这也是大多数考虑使用 Rust 的团队所处的领域（至少是联系我的那些团队），所以我认为这是在实践中真正有用的比较。如果你在编写 CLI 工具、嵌入式固件或游戏引擎，其中一些内容仍然适用，但老实说，恐怕这不是最适合你的资源。

作为背景，我之前写过关于 Go 和 Rust 的文章：2017 年的《Go 还是 Rust？选 Go。》，以及后来与 Shuttle 团队合作的《Rust vs Go：动手实践比较》，该文章用两种语言分别实现了一个小型后端服务。

## 本文你将学到

- Go 和 Rust 的重叠之处，以及它们的差异所在。
- Go 的模式如何映射到 Rust。
- 借用检查器能带来什么。
- 我建议在哪些场景保留 Go，以及在哪些场景 Rust 值得迁移成本。
- 如何渐进式地迁移 Go 服务。

## 我的立场

我先坦白：我不是 Go 的粉丝。我认为它是一门设计糟糕的语言，尽管是一门非常成功的语言。它混淆了"简便"和"简洁"，其几个核心设计权衡（到处都是 `nil`、错误处理依赖纪律规则而非类型、长期缺乏泛型）指向了一个我不认同的方向。

Go 显然对很多人来说是有效的，一份假装不是这样的指南没有帮助。所以我会尽最大努力在本指南中保持客观，而不是重新争论旧的论点。但你应该了解我的先验立场，以便你能做出校准。

另一个值得披露的先验立场：我经营一家 Rust 咨询公司；我当然有偏见！更多人使用 Rust 对我的业务有利。但我也在两种语言中都有专业工作经验，并将 Go 服务部署到了生产环境。

本指南面向那些想要诚实地、逐项对比了解迁移到 Rust 后会有什么变化的 Go 开发者。

## 最重要命令的初步了解

Go 开发者已经拥有业界最干净的工具链之一。在早期，它开创了"开箱即用"工具链的趋势，为你提供一个单一、一致的界面来构建、测试、格式化、代码检查和管理依赖。我很高兴 Rust 也效仿了这一做法，因为这是一个很好的模式。这是我最喜欢两个生态系统的地方之一。

`cargo` 内置了更多功能：

| Go 工具 | Rust 对应工具 | 说明 |
|---|---|---|
| go build | cargo build | 编译项目 |
| go run . | cargo run | 构建并运行 |
| gofmt / goimports | cargo fmt | 自动格式化，零配置 |
| go test ./... | cargo test | 测试内置于工具链 |
| go vet ./... | cargo clippy | 代码检查，Clippy 比 vet 有明显更多的规则约束 |
| go install ./cmd/foo | cargo install --path . | 安装二进制文件 |
| golangci-lint run | cargo clippy -- -D warnings | 严格检查模式 |
| go doc | cargo doc | 生成和查看 API 文档 |
| pprof | cargo flamegraph / samply | CPU 性能分析 |
| govulncheck | cargo audit | 基于公告数据库的漏洞扫描 |

最大的区别在于，在 Go 中你通常需要借助第三方工具（`golangci-lint`、`mockgen`、`air`、`goreleaser`）来填补空白。

在 Rust 中，第一方生态系统覆盖了更多开箱即用的功能。

准确地说，`cargo audit` 和性能分析工具（`cargo flamegraph`/`samply`）是社区 crate，不是 cargo 本身的一部分。

但它们连同其他外部工具（`cargo watch`/`bacon`、`cargo nextest`）只需一条命令即可安装，并且使用起来感觉就像原生工具一样，例如 `cargo install cargo-nextest` 就能让你直接使用 `cargo nextest`。

两个社区已经达成了相同的共识：一个统一的规范格式化风格（即使不完美！）比它所消除的自行车棚争论更有价值。

> Gofmt 的风格不是任何人的最爱，但 gofmt 是每个人的最爱。
> — Rob Pike，Go 箴言

`rustfmt` 也是如此：不是每个人都喜欢每个细节，但在代码审查中不再有风格争论的好处，远远超过你偶尔会有不同偏好的格式化选择。

## Go 和 Rust 的关键差异

Go 和 Rust 之间的主要差异在于你能从编译器获得什么保证，以及你对运行时行为有多大的控制权。

Go 和 Rust 都是静态类型的编译语言，都有强大的并发支持，但它们在编译器保证方面有所不同。

Go 依赖垃圾回收器、运行时竞态检测和 `if err != nil` 约定。

Rust 通过所有权、`Send`/`Sync` 和 `Result<T, E>` 将内存管理、数据竞争预防和错误处理推入类型系统。

实际的权衡是：Go 给你平缓的学习曲线、非常快的编译时间和更大的生态系统；而 Rust 给你无 GC、更严格的编译时检查和零成本抽象，代价是更陡峭的学习曲线和更慢的构建速度。

从 Go 迁移到 Rust 时，大部分变化在于检查被拉入了类型系统。Nil 处理、错误传播、数据竞争、资源生命周期、取消操作、泛型——这些都是 Go 依赖约定、工具（`go vet`、`errcheck`、`golangci-lint`、`-race`）或运行时检测来保障的东西。Rust 将它们编码为编译器直接强制执行的类型。

这是否意味着"更多的认知开销"？我对此持质疑态度。前期确实更多，是的，但也更难出错。Rust 中的 `Mutex<T>` 不仅仅是记录数据需要锁——它使锁成为访问数据的唯一途径：你调用 `.lock()`，获得一个守卫，而守卫才是给你访问内部值的东西。释放守卫，锁自动释放。不存在"我忘了加锁"的路径，因为未加锁的路径在类型中根本不存在。一旦你内化了这个模式——你会发现它到处重复出现（`Option`、`Result`、`&mut T`、`Send`/`Sync`、RAII 守卫）——Rust 就不再感觉沉重，而是感觉编译器在帮你做那些你过去用脑子做的工作。

## 重点不在运行时！

人们经常声称托管运行时对大多数后端来说"足够好了"，但我认为他们没有抓住重点。在我看来，权衡更多的是 Go 优化了快速迭代速度，而 Rust 优化了正确性。对于大多数生产工作负载来说，你对内存有更多控制权只是一个好的附带效果。这意味着你需要更少的机器来完成同样的工作，但选择 Rust 的主要原因仍然是健壮性。

## 团队考虑从 Go 迁移到 Rust 的原因

Go 开发者通常不是因为 Go"太慢"才来到 Rust 的。对于大多数后端工作负载来说，Go 已经足够快了！
