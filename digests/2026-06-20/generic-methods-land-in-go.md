# Generic Methods Land in Go

提案：Go 的泛型方法
一次观念的转变。
## 背景
为清晰起见，下文中我们使用术语"具体方法"（或在上下文明确时简称"方法"）来描述一个非接口方法——它像函数一样声明，但带有接收者；我们使用术语"接口方法"来描述接口中方法的名称和签名。
根据当前规范，具体方法是一个带有接收者的函数。但从语法上看这并不完全准确：虽然函数可以是泛型的，方法却不行。方法不能自行声明新的类型参数；但它们可以拥有泛型类型的接收者（从而拥有接收者类型的类型参数的方法局部名称）。
造成这种差异的一个原因是，我们历来将方法的主要角色视为实现接口的手段：如果允许具体方法拥有类型参数，就意味着我们也必须允许接口方法拥有类型参数。Go 不支持这样的泛型接口方法，因为我们不知道如何实现（调用）它们，或者至少不知道如何高效地实现它们。具体来说，由于 Go 不要求具体类型声明它所实现的接口，而这是一个动态属性，因此在编译时无法知道在运行时将需要具体方法的无限可能实例化中的哪些。这个不足早已为人所知，并在最初的《类型参数提案》中进行了详尽讨论。
但具体方法不仅仅是实现接口的手段。方法是与类型关联的函数，通过该类型的命名空间来访问。因此，即使方法从未实现任何接口，它们对于组织代码仍然是有用的。此外还有语法上的好处：`x.a().b().c()` 可以自然地从左到右阅读，而 `c(b(a(x)))` 则是从内到外求值。方法的这两个方面同样广为人知。
基于这些原因，Go 用户长期以来一直请求泛型方法，这个想法广受欢迎。在问题跟踪器上至少提交了两个提案：
- #49085（允许方法中使用类型参数，2021年10月，获得超过900个正面表情反应）
- #50981（为方法添加泛型，2022年2月）
到目前为止，我们一直抵制添加泛型具体方法，因为这总是意味着我们也需要泛型接口方法。Go 的常见问题解答甚至声明"我们预计 Go 永远不会添加泛型方法"。也许是时候转变观念了：具体方法本身就是一个有用的语言特性，与接口无关。如果具体方法是一个带有接收者的函数，那么泛型具体方法就是一个带有接收者的泛型函数。这样的方法不能通过接口调用，这是一个正交的方面：如果接口在语法上不能包含带类型参数的方法，那么泛型具体方法在满足该接口时自然不起作用，因为不可能存在具有匹配类型参数的接口方法。
请注意，最初的《类型参数提案》也讨论了这种替代观点，如下段所述：
"或者，我们可以决定参数化方法实际上并不实现接口，但那样的话，我们为什么需要方法就更加不清楚了。如果不考虑接口，任何参数化方法都可以实现为参数化函数。"
我们可能已经在这一点上达成了某种清晰认识：泛型具体方法本身就是有用的，即使它们不实现接口方法。
## 提案
我们提议，具体方法声明应该看起来与函数声明完全一样，只是多了接收者。具体来说，方法声明的语法应该像函数声明一样接受类型参数。函数声明的语法如下：
FunctionDecl = "func" FunctionName [ TypeParameters ] Signature [ FunctionBody ] .
方法声明的语法应从（旧）：
MethodDecl = "func" Receiver MethodName Signature [ FunctionBody ] .
改为（新）：
MethodDecl = "func" Receiver MethodName [ TypeParameters ] Signature [ FunctionBody ] .
或者，为了达到相同效果，语法可以直接表达我们口头所说的：方法（声明）是带有接收者的函数（声明），并将两个产生式合并。这减少了一个语法产生式，并强调了两者的相似性：
FunctionDecl = "func" [ Receiver ] identifier [ TypeParameters ] Signature [ FunctionBody ] .
表示泛型方法类型参数的标识符，其作用域从方法名之后开始，到方法体结束时终止。这与现有函数类型参数和方法接收者类型参数的规则一致。方法类型参数的约束可以引用与方法接收者一起声明的类型参数，因为它们在作用域内。
调用泛型具体方法按预期工作：要么根据需要提供类型实参，要么由类型推断确定任何缺失的类型实参，与泛型函数调用完全相同。
方法表达式和方法值继续按预期工作：如果方法是泛型的，则结果函数也是泛型的，并适用现有的泛型函数使用规则。
语法需要一个小调整：目前类型实参只能提供给操作数名称，即（可能带包限定的）泛型函数和类型的标识符。根据操作数的语法（旧）：
Operand = Literal | OperandName [ TypeArgs ] | "(" Expression ")" .
…
OperandName = identifier | QualifiedIdent .
TypeArgs = "[" TypeList [ "," ] "]" .
方法可以附加到任意用户定义类型上，这些类型的值可以由各种表达式表示，而不仅仅是限定标识符。为了允许泛型方法的实例化，类型实参必须从操作数的语法移到主表达式的语法中，产生更新后的产生式（新）：
Operand = Literal | OperandName | "(" Expression ")" .
PrimaryExpr = Operand |
Conversion |
MethodExpr |
PrimaryExpr Selector |
PrimaryExpr Index |
PrimaryExpr Slice |
PrimaryExpr TypeAssertion |
PrimaryExpr Arguments |
PrimaryExpr TypeArgs .
（值得注意的是，在当前实现中，类型实参已经作为主表达式的一部分被解析。这是必要的，因为像 `T[int]` 这样的实例化和像 `a[i]` 这样的索引表达式在语法上是无法区分的。在语言规范中，这两者之间的语法重叠仅仅是为了文档目的而保留的。）
接口的方法不做改变。重要的是，泛型具体方法不会与具有相同名称和签名的接口方法匹配，因为接口方法在语法上不能拥有匹配的类型参数。
泛型方法也无法通过反射（按名称或索引）访问，原因与未实例化的泛型函数不可访问相同：reflect 包没有实例化泛型值或类型的机制（而且目前不清楚如何提供这样的机制）。
以上就是本提案的全部内容。
## 示例
给定一个（非接口的）接收者基础类型 S，我们可以为其关联一个泛型方法 m：
type S struct { … }
func (*S) m[P any](x P) { … }
给定一个类型为 S 的变量 s（或类型为 *S 的值），我们可以如下调用 m：
var s S
s.m[int](42) // 显式类型实参 int
s.m(x) // 类型参数 P 从 x 推断
接收者基础类型本身可以是泛型的：
type G[P any] struct{ … }
func (*G[P]) m[Q any](x Q) { … }
调用 m 需要一个类型为 G 的实例化的变量，但在调用 m 的方式上没有其他区别。
一个带有方法 m 的接口 I：
type I interface {
m(string)
}
可以由一个适当实例化的泛型类型 G 来实现（这在当前的 Go 中已经成立）：
type G[P any] struct{ … }
func (G[P]) m(P) { … }
var g G[string]
var _ I = g // 有效，因为 G[string].m 的签名为 m(string)，与 I 的 m(string) 匹配
但 I 不被实现
