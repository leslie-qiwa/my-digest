# On Excessive nil Pointer Checks

Go 中过度的 nil 指针检查

让我们来谈谈 Go 中的 nil 指针检查。你想防止生产环境中的 panic，但这并不是从一个延迟的 `recover` 开始的。它始于防御性编程。检查你的输入，检查你的边界，并在解引用指针之前检查它们是否为 nil。

我开始在 Go 代码中看到越来越多的 nil 检查。放在正确的位置时，它们是编写安全代码所必需的。放在错误的位置时，它们就是一个信号，表明代码已经不再清楚什么可以为 nil、什么不能为 nil。我在生成的代码中更多地注意到这种模式，但这个症状并不新鲜，也不仅限于 AI。

当一个 nil 检查代价很低又能防止 panic 时，为什么不加上呢？你的本能反应可能是：稳妥起见加上吧。但这个检查也在向下一个读者传达某种信息，而且往往是错误的信息。

对依赖项的 nil 检查

考虑一个对依赖项的 nil 检查。类型 `RateLimiter` 持有一个 Redis 客户端，在第 10 到 12 行使用它之前有一个 nil 检查作为守卫：

```
1type RateLimiter struct {
2 redis *redis.Client
3}
4
5func (r *RateLimiter) Allow(ctx context.Context, req *Request) (bool, error) {
6 userID := GetUserID(req)
7 if userID == "" {
8 return false, nil
9 }
10 if r.redis != nil {
11 return r.checkLimit(ctx, userID)
12 }
13 return false, nil
14}
```

乍一看，这似乎是稳妥的做法，但你应该问自己：你为什么要用一个为 nil 的依赖项来构造 `RateLimiter`？

如果 Redis 客户端为 nil，那么错误在更早的时候——在构造时——就已经发生了。现在检查它并不能处理那个错误。更糟糕的是，它把"在构造失败的情况下继续运行"当成了一种可以接受的状态。在 Go 中，你想要的是快速失败、尽早失败。

nil 检查并不总是防御性编程。在这个例子中，它是一个信号，表明代码已经追踪不到其对象的来龙去脉了。我们不再知道这个指针来自哪里、谁负责初始化它，或者本应使 nil 不可能出现的不变量是什么。

在构造函数中对依赖项进行 nil 检查

你可能会试图通过把问题往上推一层来解决它。你现在检查 nil 并返回一个错误，以标记 nil 依赖项是一种无效状态。

```
func NewRateLimiter(client *redis.Client) (*RateLimiter, error) {
if client == nil {
return nil, errors.New("redis client is nil")
}
return &RateLimiter{
redis: client,
}, nil
}
```

这样更好，但仍然不正确。为什么？因为我们仍然允许了无效状态进入我们的系统。一个 nil 指针仍然被传递到我们的函数中，这把"是否信任该输入"的决定负担放在了本应一开始就接收到有效值的代码上。

构造函数并不是错误发生的地方。错误发生在初始化的现场：

```
redisClient, err := NewRedisClient(addr)
if err != nil {
return nil, err
}
limiter := NewRateLimiter(redisClient)
```

一旦初始化失败，我们就应该立即处理那个错误。我们不应该带着一个 nil 指针继续往下走，并迫使下一个更深的层去重新发现这个结果。这样做也消除了让速率限制器构造函数返回错误的必要性！

如果系统需要容忍存储暂时不可用，不要传播 nil，而要显式地对它建模。把它包装起来，使外层类型始终为非 nil，并在内部处理重试或降级，对外暴露可以安全调用的方法。调用方得到一个保证存在的类型，而混乱被控制在它内部。这就是你如何封装复杂性，而不是把你的整个代码库暴露给它。

这与数据库约束是同一个理念。一个 `NOT NULL` 或外键约束保证一行坏数据从一开始就不可能存在，因此每个查询都可以信任数据而无需重新检查它。你想为你的运行时值获得同样的保证，但有一个区别。数据库在每次写入时强制执行其约束。你只建立一次你的保证，这样代码的其余部分就可以依赖它，而无需重复这些检查。

静默失败的代价

当我标记出上面那样的代码时，我经常会得到一种回应，它揭示了一种关于可靠性的善意本能，大致是这样：

"我不想在这里返回一个错误，从而冒着因为我的小改动而让程序崩溃的风险。把它包在一个 nil 检查里，或者干脆记个日志，感觉更稳妥。"

这个选择感觉上像是崩溃（坏）对比继续运行（稳妥），但它实际上是响亮的失败（稳妥）对比静默的失败（坏）。一个被显式返回的错误是：

- 响亮的：你会发现它发生了。
- 即时的：你在靠近原因的地方就发现它。
- 可归因的：调用方可以把失败和导致它的那个操作联系起来。

一个被你吞掉的错误则恰恰相反。这个失败变成：

- 静默的：没有任何东西告诉你它发生了。
- 延迟的：它在之后才浮现，在更多代码已经运行之后。
- 含糊的：当你看到症状时，原始原因已经更难辨认了。

原因和症状之间的鸿沟就是代价，而且程序每多在无效状态下存活一次调用，这个鸿沟就会增长。这就是为什么修复方法不是在本地隐藏失败。修复方法是理解错误去向何方。谁调用了这个函数？错误是如何传播的？它们在哪里变成一个被拒绝的请求、一个失败的任务、一次重试、一个告警，或者一次关机？

这需要看到你当前改动集之外的地方，但那正是工作的一部分。如果返回错误会让系统超出必要地崩溃，那么问题出在错误处理的边界上。

二阶代价

一旦失败变得静默，你就不知道发生了什么，这可能会隐藏 bug。它迫使你构建基础设施来检测操作的缺失：指标、仪表盘和告警。你现在正花费工程精力去重建一个你自己扔掉的信号。每当你容忍一个不可能的或未处理的状态时，你日后都要为它付出代价去观测它。

外层 vs 内层

每个检查应该在何时发生，现在可能还不清楚。把它们看作程序执行流中的不同阶段会有所帮助。前面我写到，更深的层将不得不重新发现一个在上层已经处理过的错误。

这与操作顺序有关：外层是执行开始的地方，也是数据从外部世界进入的地方；而内层是那些外层调用最终到达的代码。你沿着调用栈往下走得越深，你就越深入内层。

在执行开始时，什么都没有保证，但你也还没有做任何事。在初始化期间，你建立起你的程序所依赖的东西。对每一个，你都要决定：这是一个必需品，还是一个有时可以不可用的东西？你的设计应该倾向于始终可用的依赖项，并尽量减少那些可能掉线的依赖项。

对请求作用域数据的 nil 检查

另一种 nil 出现在更晚的时候，一旦程序运行起来并处理工作。一个请求作用域的值——请求本身、它的字段，以及任何从它派生出来的东西——与依赖项不同。依赖项在构造时就固定了。而请求在每次调用时都从外部到来：来自一个 HTTP 处理器、一个 RPC、一个队列、一个测试辅助函数、另一个包。

在我们的 `RateLimiter` 中，对请求本身做一个 nil 检查是很有诱惑力的：

```
1func (r *RateLimiter) Allow(ctx context.Context, req *Request) (bool, error) {
2 if req == nil {
3 return false, nil
4 }
5 userID := GetUserID(req)
6 if userID == "" {
7 return false, nil
8 }
9 return r.checkLimit(ctx, userID)
10}
```

但那是在一个新的地方犯了同样的错误。请求并不是在 `Allow` 中到达的。它更早地进入了程序，在传输边界处——HTTP 处理器、RPC 分发、队列消费者等等——从那以后它就一直在我们的代码中穿行。

当 `Allow` 运行时，请求已经是内层数据了。在这里检查它意味着一个深层函数正在重新校验某个外层本应已经保证的东西，而这正是我们针对依赖项所拒绝的做法。它扩散了不确定性。这个检查属于边界，在那里，不受信任的字节第一次变成一个 *
