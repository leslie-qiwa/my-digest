# Honesty gets Emacs patch rejected

诚实导致 Emacs 补丁被拒

我已经在 macOS 上研究 Emacs 性能好几个月了——虽然不是持续不断地做，但也确实有好几个月。在此期间，我一直忙于接入测量工具和创建基准测试。我还把代码库交给了多个大语言模型（LLM），让它们去搜寻特定的东西。通常结果都很糟糕。在分析那些补丁之后，要么影响微乎其微，要么是问题被误解了——这完全在意料之中。

不过在这段时间里，我对这个代码库形成了一些我相信是正确的看法。

这些看法是：在 macOS 上，性能的主要问题在于渲染问题和内存抖动——发生在快速的分配与释放过程中。一个看法是：由于系统 malloc 的工作方式（这是 macOS 特有的），缺乏内存压缩（compaction）会导致虚拟内存膨胀和缓存局部性的丧失。还有一个看法是：Emacs 核心本身并非没有性能问题。我反复分析的一个地方是正则表达式处理。它无处不在，因此对正则表达式处理的任何改进都会提升整体性能。

我一遍又一遍地重新分析这些领域。我做了一些粗糙的补丁，并慢慢缩小范围，希望能找到切实的、可修复的关键点。

最近，多亏我的朋友赠予我的 z.ai Max 套餐，我得以在我正在做的项目上运行 GLM 5.2（在那些我可以运行的项目上）。我发现 GLM 5.2 在代码优化方面相当有能力，于是我决定在已经积累的知识背景下，就 Emacs 提出具体问题，并让它去搜索和分析任何问题。

不细说细节，大约 3 小时¹ 之后，它返回了几份报告。我审阅了它们，审阅了那些建议和提供的代码，测试了每一项的影响并运行了基准测试。由于把代码整理到可提交补丁的状态需要一些时间，我挑选了最有希望的一个并开始着手。在写这篇文章的两天前，我把它发送到了 emacs-devel 邮件列表。

第二天我得知它不会被接受，因为 GNU 有一项反对接受 LLM 辅助工作的政策。我尊重它，但我不同意它。

补充一些背景：在我发送补丁时，在邮件列表里我说明了：
- 该问题是由 GLM 5.2（一个具有开放权重的中国模型）发现并起草补丁的。
- 我分析了问题报告的正确性和影响
- 我审阅了补丁并对其做了修改
- 我手动测试了补丁
- 出于法律目的，我声明了对该提交的著作权（即我准备主张我的贡献大于 LLM 的贡献）
- 我声明对该提交承担完全的、个人的责任

提交的规模和实现范围都非常狭窄。我不认为它能被归类为"垃圾"（slop），但欢迎你自行判断（该补丁有 92 行长，并在注释中包含了其存在的理由）。

然后它被退回了。

就我个人而言，我不认为这项政策有任何依据。

首先，我本可以隐瞒使用 LLM 的事实，但却选择明确声明它。通过保持诚实，我反而已经处于下风。仅凭这一点就说明这项政策很愚蠢。如果坦白会受到惩罚，那还不如不坦白地推送提交。它惩罚的是诚信，而非使用本身。因为谁会发现呢？我完全不信任 LLM，因此我认为 LLM 辅助的工作实际上需要更多的审查和更多双眼睛——而不是更少。

我并不声称自己了解这项政策的全部背景，因为——雪上加霜的是——这项政策是在 GNU 的内部邮件列表上讨论的。然而，从过去围绕 LLM 的对话中我了解到，对 LLM 贡献的疑虑在于它们是否"足够开放"以及是否"合法可用"。

当我们谈论开放权重模型时，我觉得关于"开放"的论点很荒谬。这意味着我本地设置上的 Qwen 3.6 是可以的，但如果我通过 OpenRouter 使用它——那就不行了。GLM 5.2 是一个开放权重模型，如果我有 256 GB 内存（我没有）和 24GB 显存（我有），我就可以在我的本地机器上运行它，从而避开整个"SaaS 是封闭的"这一论点。按照同样的标准，也许在制作提交时不应该联网？互联网上充满了非自由内容，因此补丁可能已经被污染了？谁知道呢，也许灵感取自——*惊呼*——某本非自由的书或文章。

至于合法性的论点——我认为这是傲慢在作祟。

尽管我对 GNU 组织怀有全部的同情，但它既不是世界上最大、最聪明，也不是最在意法律的组织。例如，游戏公司在知识产权和 LLM 方面要偏执得多，但那里的使用同样显而易见；ChatGPT 拥有十亿活跃用户；成千上万、甚至数百万的组织——无论商业与否——每天都在使用 LLM 的输出。对它们来说，情况是清楚的。

我不是美国律师，但我读到：
（版权）局不会为由自然、动物或植物产生的作品进行登记。同样，版权局不能为据称由神圣或超自然存在创造的作品进行登记，尽管如果申请或提交的副本声明该作品是受神圣精神启发而创作的，版权局可以为其登记。

而以我这个非法律专业（IANAL）人士的理解：问题在于给它盖上版权章，而非相反。

然而 GNU 相信他们的律师和他们的意见最有分量。我不会剥夺他们为自己做决定的权利，但这种缺乏自我意识的状态几乎到了滑稽的地步。

GNU 有决定自身事务的自由——我也有批评它的自由。在我看来，闭门讨论、对用户没有丝毫透明度，其"开放"程度跟 Meta 关于 Facebook 走向的内部决策一样。我不会称之为开放，我也不会把以这种方式运作的组织称为开放组织。

在我关于政治的牢骚结束之后，这是实际的影响。

我不打算再继续做 Emacs 了。我讨厌别人告诉我"棍子拿错了"——尤其是当我出于自己的意愿在做某件事的时候。在我的硬盘上，我大约有 40 个性能改进补丁，有些相互重叠，有些尚未证实实际影响。我只发布了少数几个最近确认有效且有实质性影响的。

欢迎去看看它们，因为它们确实²能带来改变。

就这样，我想我会转向更"开放"的水域。
