# Migrating from Proxmox to NixOS and Incus

> 注：原文为一篇较长的署名博客文章。为避免整篇复制受版权保护的内容，以下提供按章节组织的中文要点编译，而非逐字全文翻译。

**我全面转向 Nix：从 Proxmox 到 NixOS + Incus**
*为什么我用声明式文本文件取代了点击式虚拟化平台*

作者正式下线了运行多年的 Proxmox 集群（从单台 NUC 起步、扩展到多节点），将全部环境迁移到运行 Incus 的 NixOS。

**从怀疑者到信徒**
作者最初厌恶 Nix 的语言和语法，已有自己的 dotfiles 管理方式（Dotbot 做符号链接、自写的 dotbins 管理二进制），在 Mac 上仅用 nix-darwin 管理 Homebrew 包和应用设置。真正的转折发生在购入游戏 PC 之后：Pop!_OS 上的 NVIDIA 驱动问题需要靠不可复现的命令式操作修复，一次更新驱动后陷入 GRUB 启动循环。改装 NixOS 后，原子更新带来的可复现性令其折服——把系统迁移到新硬盘时无需克隆，只需对全新安装应用同一份 Nix 配置，结果完全一致。

**命令式系统的摩擦**
Proxmox 优秀且降低了入门门槛，但本质是图形界面优先、围绕点击操作构建；即便用 Terraform/Ansible 自动化也常感别扭，状态漂移真实存在。引入 AI 智能体后问题更严重：智能体「YOLO 模式」可能执行上百条命令式操作，即便成功也会留下无人能复现的未定义状态。硬件层面亦如此——HP EliteDesk 上 Intel I219-LM 网卡硬件卸载会挂起，在 NixOS 中修复变成了带注释、引用论坛出处的 systemd 服务，重装即自动生效；Intel NUC 则借助 NixOS 让宿主直接运行 Kodi（原生硬件加速与视频输出），同时后台用 incus 托管容器，无需在「媒体播放器」与「可调试虚拟化宿主」之间二选一。更深层的区别在于：Proxmox/TrueNAS 是「设备型」系统，鼓励用户别在宿主上随意操作；而 NixOS 的宿主完全属于自己，声明式状态使一切显式可复现，破坏后可秒级恢复。

**智能体的放大效应**
在 AI 智能体执行任务的时代，CLI 优先、声明式系统是王道——智能体无法可靠地「点按钮」配置 VLAN 或扩容磁盘，它需要文本和确定性。迁移到 NixOS 后整套基础设施由文本文件定义，智能体可读取、理解乃至安全修改。例如要「在 9000 端口部署 Faster Whisper API 并暴露到局域网」，只需写一个 systemd 服务定义并加上 `networking.firewall.allowedTCPPorts = [ 9000 ];`，智能体还能通过 git diff 或当前配置验证改动。作者坦言其管理 9 台机器、结构清晰的配置几乎没有手写——全部由 AI 智能体跨多次会话完成、反复重构，确保 PC、NUC、HP 共享通用模块又保留各自特性。

**架构：Incus 与仿真**
作者仍希望有纯「Nix」方式管理有状态的 LXC 容器与虚拟机（nixos-containers、microvm.nix 等项目运维成熟度或热迁移能力不足）。Incus（LXD 的社区分支）恰好填补空缺：宿主用 NixOS 做「牛群」式管理，同时以稳定可控的方式运行旧 Ubuntu 容器、Home Assistant 等「宠物」遗留负载，且完全可由简洁 CLI 控制，契合智能体工作流。作者此前已把多数遗留 LXC 服务迁到 VM 内的声明式 Docker Compose，唯一的「钉子户」Home Assistant OS 也发现可在 NixOS 上的 Incus 中以完整 VM 轻松运行——不过是又一个 QEMU 进程。作者还用 Incus 虚拟机复刻物理机的完全相同配置，在关停最后一台 Proxmox 宿主前便已确信完整配置可用。
