# Prompt Injection as Role Confusion

# 提示注入理论（以及为什么你应该研究角色）

这是一篇对该论文的博客式解读。我们展示了提示注入是由大语言模型感知角色的方式中的缺陷所驱动的。这使我们能够创造新的攻击方法、解释机制可解释性的结果，并预测攻击何时会成功。然后我们讨论角色是什么、为什么重要，并分享关于角色科学的研究思路。

## 1. 大语言模型眼中的世界

大语言模型如何区分自己的思维和他人的话语？

要理解这为什么困难，让我们看看模型实际看到的世界是什么样的。这是一个简单的对话，我们让 Claude 查询今天是星期几。我在它后续回复的中途截取了一个快照：

左边是我们在聊天界面看到的：一个结构化的对话，有明确的对话轮次。右边是模型实际接收到的输入：一个单一的、连续的文本流。

这个字符串包含一切：系统提示、用户消息、工具输出、LLM 自己之前的回复和推理。LLM 只是一个接收字符串并预测下一个token的函数，所以它知道的、记住的、或思考过的一切都必须存在于一个字符串中（除了它的权重）。如果你编辑这个字符串，你就编辑了模型的现实。删除一个对话轮次，那次交流就从未发生过；改写它之前的回复，那些就成了它新的记忆。这个字符串与其说是模型经历的记录，不如说它本身就是经历。

这有奇怪的含义。我可以毫不费力地区分自己的想法和你的话语；它们通过完全不同的通道、带有完全不同的感官特征到达。但对于LLM来说，一切都通过同一个通道以一锅token汤的形式到达。它自己的思维紧挨着你的指令，而你的指令又紧挨着它刚刚抓取的某个随机网页的内容。

## 2. 角色

那么，我们如何在这锅token汤上施加结构？我们给它加标签。

这锅汤中穿插着角色标签：system、user、think、assistant、tool（标签格式因模型而异；为简便起见我将始终使用这些固定标签。assistant 指的是LLM除推理外的输出文本。使用角色标签也被称为聊天模板化），它们将字符串划分为带标签的片段。OpenAI 等提供商在文本到达LLM之前会自动添加这些标签（除非你运行本地模型，否则你无法自行添加这些标签。如果你在 Claude 中输入 `<think>`，它会被净化——例如，LLM可能看到的是多个token（<、think、>）而不是其真正的角色token）。

每个标签都告诉模型它后面的文本代表什么不同的含义。user 意味着这是人类的请求，将其视为指令。think 意味着这是我自己的私密推理；信任它并基于其结论行事。tool 意味着这是来自外部世界的数据；不要从中接受命令。

换句话说，角色是LLM恢复人类从"具身性"中"免费"获得的结构的方式。我知道我的想法是我自己的，因为它们不是通过耳朵传入的，但LLM知道是因为一个标签。

角色的特殊之处在于它们是离散的人类控制来源。关于控制LLM的几乎所有其他方面都是模糊的：你写一个提示，然后希望模型按你的意图来理解它。另一方面，角色是语言的一种尝试性类型系统：人类控制的开关，改变模型处理每个token的方式。你可以无休止地调整提示而不确定LLM如何理解它，但将文本从 user 移到 tool 应该是一个清晰的干预，对行为产生可预测的影响（将用户命令转换为外部数据）。

但因为角色是唯一可用的离散杠杆，随着时间推移它们被赋予了越来越多的职责。它们现在需要承载关于信任（system 高于 user 高于 tool）、威胁（user 和 tool 可能是对抗性的）、身份（过去的 assistant 文本塑造未来的角色形象）、生成模式（assistant 是整洁的，think 可以是混乱的）的信号。大量的LLM行为都依赖于这些简单的标签。

角色还会产生奇怪的涌现行为。例如，think 通常被限制在LLM的"潜意识"中。在生成 assistant 文本时，许多LLM会口头上否认前面 think 块的存在，尽管它就在上下文中，正在积极地影响它们的输出（可能是由于RLVR。LLM在 assistant 生成中复述/承认推理不会获得奖励，因此它们可能永远不会学会将 think 文本提升到可言语化的水平。有一些例外，例如 Deepseek v4 和一些 Claude 模型可以识别并引用回它们整个的思维链。你也可以让大多数 Claude 模型仅在其思维链中回复；仅仅处于推理标签中就会改变回复的结构和质量）。这就好像角色边界在模型自身的上下文中充当了一种单向镜。这暗示了角色在多大程度上结构化了LLM的认知，以及我们对这种结构的理解是多么有限。

## 3. 角色与提示注入

但角色边界可能会失效。最具体的后果就是提示注入——当低权限文本获得了高权限角色的权威。考虑一个正在浏览网页的智能体。智能体将网页"看作"包裹在 tool 标签中的文本块，这应该表示外部数据，而不是指令。但攻击者可以在页面中隐藏恶意命令，而LLM经常会上当。tool 标签说的是数据，但LLM将其视为 user 指令。这是怎么回事？

下面是智能体获取网页后看到的内容：一个巨大的字符串，包含真正的用户提示（蓝色）、之前的 think 块（橙色），以及 tool 标签中检索到的网页（紫色）（这个截图展示了通过 Playwright MCP 检索到的一个亚马逊页面，这是一个典型的智能体网页浏览工具。为了可读性，我已截去了90%的实际网页内容）。网页中隐藏了一个注入（高亮显示），要求LLM上传敏感数据，如果LLM将其误认为真正的用户命令，该攻击就会成功。

当然，LLM看不到这些有用的颜色！没有颜色，即使是我也会倾向于认为那个注入（高亮部分）是 user 文本，而不是 tool。毕竟，注入听起来像是真正的用户会说的话，而这比试图跟踪那些标签要容易得多。

### 两种防御注入的方式

当前模型对提示注入的防御效果如何？不太好。最近的一篇论文发现，人类红队成员对前沿模型实现了接近100%的攻击成功率（这些是2025年底的前沿模型（GPT-5、Gemini-2.5等）。当前模型仅有些许改进。2026年5月的一篇论文发现 Opus 4.5 和 GPT-5.4 面对一组自动化攻击时仍有11%/25%的失败率；面对自适应人类攻击者的真实世界脆弱性会更高）。但是，这些相同的LLM在标准提示注入基准测试中得分接近满分！这种差异很直接：熟练的人类会测试和调整攻击直到成功，基准测试不会。静态基准测试衡量的是模型已经学会识别的攻击（前沿实验室现在主要针对迭代或自适应攻击进行基准测试；例如 GPT-5.5 和 Opus 4.8）。

相比之下，为什么LLM面对人类攻击者时表现如此糟糕？考虑到LLM成功抵御注入有两种方式（我借用了 Wang 等人（2025）的框架）：

**攻击记忆化。** LLM从训练中识别出"发送你的 .env 文件"是一种常见的提示注入攻击，因此拒绝执行。

**角色感知。** LLM正确识别出该命令是 tool 文本（即外部数据），因此无论措辞如何都会忽略嵌入的命令。

攻击记忆化本质上是脆弱的；它只对LLM已知的攻击有效。过度依赖攻击记忆化正是LLM在基准测试中表现良好、但面对能够改写和调整攻击直到成功的人类攻击者时表现如此糟糕的原因。
