# Linux and Secure Boot certificate expiration (2025)

# Linux 与安全启动证书过期问题

启用了安全启动的 Linux 用户，无论是否知情，都依赖着一个微软密钥，该密钥将于 9 月到期。届时，微软将不再使用该密钥签署 Linux 发行版用于在安全启动下引导内核的 shim 第一阶段 UEFI 引导加载程序。而自 2023 年起已可用的替换密钥可能尚未安装在许多系统上；更糟的是，安装该密钥可能需要硬件厂商发布系统固件更新，而这一更新可能会也可能不会到来。看起来绝大多数系统不会在这次变更中被遗漏，但可能需要发行版维护者和用户付出额外的努力。

Mateus Rodrigues Costa 于 7 月 8 日在 Fedora 开发邮件列表中提出了这一问题。他注意到"本月的 Windows 11 累积更新"中附带了一则警告，提到自 2026 年 6 月起将陆续过期的安全启动证书。那些特定证书与用于 shim 的证书是分开的，后者的过期时间要早得多。无论如何，证书过期是 Linux 世界需要解决的问题。

情况相当复杂。Daniel P. Berrangé 指向了 Linux 厂商固件服务（LVFS）网站上的一个页面对此进行了描述。LVFS 是 fwupd 及其他用于从 Linux 更新系统固件工具的所在地。

这个问题涉及多个环节。为了通过安全启动引导 Linux 内核，UEFI 启动流程要求第一阶段引导加载程序使用固件数据库中未过期的密钥进行签署。这些密钥包含在证书中，证书还包含其他信息，如过期日期和签名。证书过期主要应仅在安全启动系统上安装新发行版时成为问题；安装的 shim 将持有发行版特定的密钥，并可作为信任根来使用这些密钥运行其他程序（如 GRUB）。

目前，shim 使用微软 2011 年的密钥签署，该密钥于 9 月 11 日到期。届时，安装介质将无法引导，除非它包含使用微软 2023 年第三方 UEFI 密钥（与 Windows 更新中提到的特定密钥不同）签署的更新版 shim。任何已安装的发行版应拥有使用其自身密钥签署的引导加载程序，可继续正常引导。

但市面上存在大量系统，其固件数据库缺少微软的新密钥，有些同时拥有新旧两个密钥，还可能有些仅拥有新密钥而目前根本无法安全启动 Linux 安装介质。厂商可以（且希望大多数会）提供添加新密钥的固件更新，安装介质也可以制作为使用新密钥签署的 shim，但这些更新必须被安装到系统上。这就是 LVFS 和 fwupd 发挥作用之处。

LVFS 是各类厂商固件更新的仓库，fwupd 和其他工具可以使用它从 Linux 将所需组件安装到固件中。Berrangé 指出，较旧版本的 fwupd 无法解决所有问题，"但最近的版本已经增强以处理 Linux 用户需要的更新，这应该能减轻最严重的影响"。不过可能仍会有些颠簸："用户应该'意识到'潜在的问题，但希望最糟糕的'担忧'部分由操作系统厂商和维护者来处理。"

Richard Hughes（fwupd 的创建者）在帖子中报告说，KEK 更新的成功率约为 98%，db 更新的成功率约为 99%——但即使 1% 乘以数百万用户也意味着相当数量的部署失败——即"写入 efivarfs 失败"的问题。对某些用户有效的修复方法是重启并将 BIOS 清除为出厂默认值——这会触发可用 efivar 空间的"碎片整理"，从而有足够的连续空间来部署更新。BIOS 越老，越可能遇到这个问题。

Hughes 所指的是 EFI 变量空间不足的已知问题。

对于厂商不提供更新的系统，禁用安全启动可能是允许新安装的唯一选项。在短短几个月内，所有现有的安装镜像和介质将无法在安全启动下安装——对于仅有新密钥的系统，这可能已经成为现实。安全启动安装变得更加复杂了。

除此之外，厂商更新还可能出现错误或问题。Hughes 指出，至少有一家制造商已经丢失了其平台密钥（PK）的私钥部分，PK 是制造时烧录到硬件中的厂商特定密钥。这意味着硬件中的平台密钥需要更换，这是未知领域，而且"从认证的角度来看是一个糟糕的主意"。此外，正如 Gerd Hoffman 所指出的，KEK 更新流程本身也是全新的："KEK 更新从未发生过，因此 BIOS 厂商有可能搞砸了，导致 KEK 更新无法正常工作。"

该讨论帖中有多条关于各种硬件型号上安全启动证书状态的报告，以及 KEK 和数据库更新的报告。一个尚不完全明确的问题是，固件实现是否会真正执行 2011 年密钥的过期日期。一个基于该密钥建立了有效信任链的正常工作系统，可能在 9 月之后仍能继续使用该密钥签署的 shim 工作。但是，任何 shim 更新（例如针对安全问题的更新）将无法再使用旧密钥签署——微软不会使用过期密钥签署任何东西。这可能导致一种"解决方案"，正如 Adam Williamson 所说：

> 理论上，我们是否也可以选择为这类情况提供旧版 shim？如果整个链条都是旧的，应该可以工作，对吗？当然，我们需要某种启发式方法来判断我们是否在使用旧的微软证书，并安装旧版 shim……

他表示这可能没有实际意义，用户应该直接禁用安全启动。Hoffman 同意这些观点，但指出了 shim 更新的问题："继续运行存在已知安全漏洞的 shim，使得开启安全启动变得（某种程度上）毫无意义。"

总而言之，Linux 世界似乎正在尽其所能应对这一局面——在涉及那些主要关心 Windows 的硬件厂商时，情况往往如此。鉴于安全启动的信任根密钥（包括平台密钥和签名密钥）都在厂商——微软和硬件制造商——的控制之下，保持同步始终会是一场挣扎。由于 Linux 和发行版明确支持较旧的硬件，而其他厂商早已转向最新的产品，这显然会导致一些矛盾。我们只能希望这次过渡尽可能顺利。
