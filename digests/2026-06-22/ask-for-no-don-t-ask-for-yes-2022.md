# Ask for no, don't ask for yes (2022)

我认为保持行动导向的偏好非常重要。和其他事情一样，这是可以养成习惯的。向前推进能让你取得进展。不知道你是否有过这样的经历，但我过去常常因为不知道正确的方向而陷入停滞。向前迈出哪怕最小的一步，都有助于打破这种僵局。

我喜欢的一个习惯是：寻求"不"，而非寻求"是"。请注意，这基于我在小公司（少于200名员工）的经验，我的大部分经历都是在这类公司。我不确定这在大公司、非营利组织或政府机构中是否同样适用。

当你想做某件事，觉得它属于你的职责范围，但又想要一些安心感或让老板知道你在做什么时，人们通常会去征求许可。别这样做。不要请求"同意"。相反，给对方一个说"不"的机会，但附上一个截止日期。

让我们看看这是怎么运作的。

假设我想设置一个新的 GitHub Action，我认为它能真正提高我们的软件质量。这不是心血来潮，我做过一些研究并在本地测试过了。我甚至可能问过一位前同事他们是如何使用这个 GitHub Action 的。

但我不太确定。我想让老板知道我将要修改代码仓库。

我可以说："嘿，老板，我们能安装 Action X 吗？它能帮助解决我们一直遇到的 XYZ 问题。"

如果你的老板很忙（大多数人的老板都很忙），这需要他们花一些精力来说"是"。

他们会想要审视 XYZ 问题，思考 X 如何解决它，也许还要做一些思考或优先级排序，看看这与其他工作如何配合。或者他们可能希望你分享你了解的信息。这件事可能会从他们的待办事项中滑落。你可能不得不提醒他们好几次才能等到他们说"是"。对你来说这可能是个更紧迫的问题。

现在，让我们换一种方式。"嘿，老板，我打算安装 Action X，它应该能解决我们一直遇到的 XYZ 问题。除非收到你的不同意见，我会在周一处理这件事。"

你感受到语气的变化了吗？

你在表达（虽然没有明说）你"搞定了"，你会处理这个问题。老板仍然可以介入发表意见，但不是必须的。如果他们忘了这件事或其他问题冒出来，你仍然可以继续推进。这让你能持续前进、解决问题，同时让老板保持知情，并允许他们在足够重要时发表意见。

你也可以在一群人中使用这种方法。

顺便说一下，截止日期也至关重要。假设今天是1月15日，在其他条件相同且需要回复的情况下，你会更快回复哪一个？

- "我打算做任务 X。"
- "我打算在1月17日做任务 X。"
- "我打算在2月15日做任务 X。"

我会回复第二个，因为它有一个近在眼前的截止日期。我认为大多数人都是这样的。

再次强调，这种方法适用于你认为属于你职责范围、但又想告知老板的问题。当你想给对方反馈的机会，但又对行动方案有足够信心、并不真正需要反馈时，这种方式非常有效。
